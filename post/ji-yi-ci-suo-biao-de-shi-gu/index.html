记一次锁表的事故

2021-11-17 4 min read

前言

客户反馈说，测试环境数据同步出现异常，接口超时无响应

事故排查

  1. 检查测试环境接口，确实存在超时的情况，然后检查服务器日志， 发现一条sql语句一直超时，简单排除了网络和数据库的问题，初步推断是这张表被锁住了
  2. 连接数据库，查看数据库表锁，发现确实存在锁住的情况而且长时间没有释放。

bug复现

  1. 定义一个接口，这个接口使用spring的事务
@RestController
@RequestMapping("/lock")
public class SQLController {
    @Autowired
    private MaxNoService maxNoService;
    @Autowired
    private PlatformTransactionManager transactionManager;

    @GetMapping("/test")
    @Transactional(rollbackFor = Exception.class)
    public String lock() {
        maxNoService.getMaxID("test", "ID", 1);
        DefaultTransactionDefinition transactionDefinition = new DefaultTransactionDefinition();
        transactionDefinition.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRES_NEW);
        TransactionStatus transactionStatus = transactionManager.getTransaction(transactionDefinition);
        try {
            maxNoService.getMaxID("test", "ID", 1);
            // 其他表的操作
            transactionManager.commit(transactionStatus);
        } catch (Exception e) {
            transactionManager.rollback(transactionStatus);
        }
        return "over";
    }

}
  1. 定义数据库处理业务
@Service
public class MaxNoService {

    @Autowired
    private ZdmaxnoRepository mapper;

    @Transactional(isolation = Isolation.READ_COMMITTED)
    public long getMaxID(String noType, String subType, int size) {
        if (size < 1) {
            size = 1;
        }
        Long maxValue = mapper.selectForUpdate(noType, subType);
        if (maxValue != null) {
            long t = maxValue.longValue() + size;
            mapper.updateMaxNo(t, noType, subType);
            return t;
        } else {
            Zdmaxno maxno = new Zdmaxno();
            PeopleKey peopleKey = new PeopleKey();
            peopleKey.setNo_sub_type(subType);
            peopleKey.setNo_type(noType);
            maxno.setPeopleKey(peopleKey);
            maxno.setNo_max_value((long) size);
            maxno.setLength(10L);
            mapper.save(maxno);
            return size;
        }
    }
}

public interface ZdmaxnoRepository extends JpaRepository<Zdmaxno, PeopleKey> {
    @Query("select no_max_value from zdmaxno" +
            "    where no_type = ?1" +
            "    and no_sub_type = ?2")
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    Long selectForUpdate(String noType, String subType);

    @Query("update zdmaxno set no_max_value = ?1" +
            "    where no_type= ?2" +
            "    and no_sub_type= ?3")
    @Modifying
    Integer updateMaxNo(long t, String noType, String subType);
}

这里的 Lock 就是 for update,为这条记录的写入加锁

原因分析

  1. for update 给表记录加上了写入锁，这样的话只有这个加锁的连接才可以写入记录，其他连接想要操作这条记录只能等待。
  2. spring的事务在默认情况下，一个事务共用一个连接，首先第一个事务获取为这张表加上了锁，因为事务还未提交，所以锁还未释放。然后通过手动事务的方式创建了一个新的事务，这个事务使用新的连接去操作这张表，但是因为这张表已经被锁住，它就只能阻塞等待，因为它的阻塞等待，上一个事务无法提交，所以只能等待超时抛出异常，所有事务都处理失败。

解决方法

  1. 因为这里的业务的处理逻辑是想获取全局唯一的最大ID，所以完全可以把手动提交事务中最大ID的获取提取到第一个事务中，这样就不会死锁了
  2. 将子事务的传播行为改成 PROPAGATION_REQUIRED,这样当前事务就会加入到前一个事务，但是这里业务考虑的是希望现在直接提交，而不是跟随第一个事务一起提交，所以这里不合适
